B BIDS-Manager comprehensive GUI proposals · v2

Five comprehensive GUI proposals — full functional surface, real data

Each proposal renders the complete BIDS-Manager surface: the Converter view (raw input picker, file system tree, modality-filter tree with checkboxes, inspection table, entity editor, BIDS preview tree, output picker, status bar) plus the Editor view (validation-aware BIDS browser, type-routed file viewer for NIfTI / sidecar JSON / TSV / DICOM, file-and-dataset validation panel). Switch between Converter and Editor with the view sub-tabs under each proposal.

All five share the same engine assumption — BIDS schema as the rules engine, every form generated from it, every keystroke validated. They differ in presentation.

Press 1–5 for proposals · ⌘/Ctrl+B for sub-tabs Real data: neuroimaging_unit_new · OL_0001/0002/0003 · Siemens Prisma 3T

1 What you're looking at

  1. 1Path bars at top. Raw input + BIDS output, both editable inline. Green check when the path resolves; the rest of the GUI only activates when both are set.
  2. 2Raw FS tree (col 1). Lives mirror of the input directory. Skipped sequences (localizers, reports) appear struck-through; clickable to drill into a series.
  3. 3Modality filter tree (col 2). Three-state checkboxes (off / partial / on). Toggling a node toggles every descendant. Always-exclude patterns sit at the bottom, persisted across sessions.
  4. 4Inspection table (col 3). One row per included recording. Inline status badge, classifier confidence, predicted basename. Selecting a row populates col 4.
  5. 5Properties panel (col 4). Datatype + suffix dropdowns at top, then schema-generated entity form, predicted path, validation, "Why this name?" provenance.
  6. 6Bottom tabs. BIDS preview tree (mirror of what Run will create), conversion log, conflict resolver, statistics. Drag-resize the divider.
  7. 7Editor view. Same shell. Validation-aware BIDS tree on the left (green/orange/red). Center is type-routed: NIfTI viewer for .nii(.gz), schema-driven sidecar form for .json, table for .tsv, header dump for DICOM. Right side aggregates validation messages at file/folder/dataset scopes with per-rule fix buttons.

2 User journey

  1. Pick raw + BIDS paths. Top toolbar Run is greyed until both resolve.
  2. Scan. The classifier chain runs (BidsGuess → mne → sequence dict). Inspection table fills row-by-row with confidence values.
  3. Triage. Click 1 error to filter the table. Or open the Conflicts tab if studies / subjects collide.
  4. Fix. Type into the entity form on the right; basename + validation badges update on every keystroke. Fix the missing task on the broken row.
  5. Bulk. Multi-select 6 fmap rows, change run values once; all six basenames update.
  6. Run. Toolbar Run opens a dry-run diff against the BIDS preview tree; confirm; per-row progress in the table.
  7. Switch to Editor. Validation-aware BIDS tree. Click any red folder; right pane shows the failing rules with one-click fixes (Auto-fill, Repopulate, Set "mb"). Click the JSON sidecar; schema-driven form opens with REQUIRED red-bar / RECOMMENDED orange-bar / OPTIONAL grey-bar / DEPRECATED struck-through.

Strengths

  • All four panels visible at once — power users see everything.
  • Familiar Xcode/Unity/Final Cut shape; muscle memory transfers.
  • Modality filter tree's three-state checkboxes give exact selection without typing filters.
  • Editor's three-pane layout cleanly handles file viewer + validation in parallel.
  • Lowest porting cost from current BIDS-Manager/ shape.

Weaknesses

  • Four columns compete for screen width; hostile on 13" laptops.
  • First-time users see a wall of UI; needs onboarding.
  • The grid hides the *narrative* of an acquisition (how a session unfolded).

Notes

  • Both views need QTableView + custom model (not QTableWidget) for 50k-row scale.
  • Type-routed viewer is the v0.2.5 MetadataViewer.load_file pattern — port verbatim, schema-coloring is the new layer.
  • Modality filter tree exists in v0.2.5 (Specific tab); checkbox tri-state already works.

Best for: users who already know BIDS and want a fast, dense, keyboard-driven workflow over real datasets like neuroimaging_unit_new. Lowest implementation risk; closest to the current tool's shape.

1 What you're looking at

  1. 1Six-step stepper. Every screen shows the global progress; you can jump back. "Next" is disabled until phase invariants hold (Edit phase requires zero errors).
  2. 2Filter rail (left). Same modality-filter tree as Inspector — by subject, datatype, status — with three-state checkboxes.
  3. 3Per-subject expandable cards. Each card lists every recording with its raw → BIDS mapping. Errors expand inline with the entity editor right there.
  4. 4Live BIDS preview (right rail). Updates as the user edits. Errors propagate up — broken filename gets the red badge.
  5. 5Audit phase = Editor. Same rails after conversion. Center is type-routed: NIfTI tri-planar viewer for .nii.gz, schema-driven JSON form for .json, validation messages with one-click fixes on the right.

2 User journey

  1. Scan phase: pick raw + output paths, watch the inventory roll in.
  2. Review: auto-classification with confidence numbers; user accepts or marks for editing.
  3. Edit (shown): error rows expand inline; type the missing task; preview tree on the right turns green.
  4. Validate: cross-row + dataset-level checks; phase complete when zero errors remain.
  5. Convert: per-row progress; failed rows stay on screen with retry.
  6. Audit (shown): NIfTI viewer + sidecar form + dataset validation panel. One-click fixes for remaining warnings.

Strengths

  • First-time users finish a conversion without reading docs.
  • Forces a validation pass; impossible to skip past errors.
  • Per-subject card view keeps the dataset-as-narrative mental model.
  • Audit phase doubles as the post-conversion editor.

Weaknesses

  • Power users hate wizards on big datasets.
  • Stepper takes vertical space at every screen.
  • Going back across phases for "just one fix" feels slow.

Notes

  • Could ship as the default mode for new projects, with an "expert" toggle to Inspector.
  • The expandable card with inline entity editor is reusable in other proposals.
  • The Audit phase shares 90% of its code with the Inspector Editor view.

Best for: students, lab newcomers, training scenarios. Not the daily-driver shape for a power user with 200-subject studies, but a great first-run mode.

1 What you're looking at

  1. 1Combined sidebar (left). Three stacked sections: filter (with tri-state checkboxes), raw FS tree, and BIDS preview tree. The same rail is reused in the Editor view but loads the post-conversion BIDS structure with validation badges.
  2. 2Subject cards (center). One card per subject. Acquisitions render as compact rows with modality icons (🧠 MRI, 📈 physio, 📡 EEG, 🪄 MEG). Status badge per row + per card. Selecting a card highlights the card and feeds the right pane.
  3. 3Detail pane (right). Selected recording's identity inference + classification rationale (the "why this name?") + entity editor + predicted path + validation. All the Inspector properties panel content, but with the provenance promoted up top.
  4. 4⌘K palette. Top search bar is also a command palette: "go to sub-002", "convert all", "validate dataset", "show fmap with empty IntendedFor". Plain English.
  5. 5Editor view. Same three-rail shell. Left rail switches to BIDS-tree-with-validation-badges. Center is the type-routed viewer (NIfTI tri-planar with vol slider for 4D, JSON form, TSV table). Right rail is the validation panel.

2 User journey

  1. Drop folder. Cards appear, one per detected subject. Modality icons + status badges roll in.
  2. Eye-scan. Red badges immediately visible. The broken sub-002 card stands out without filtering.
  3. Click broken card. Right detail pane opens with raw evidence + entity editor.
  4. Type the missing entity. Predicted path updates; badge flips to green; preview tree on the left rail recolors.
  5. ⌘K → "convert". Cards switch to per-row progress overlays.
  6. Switch to Editor. Same shell, now showing the converted dataset. Click any red folder; right pane shows fix actions.

Strengths

  • Dataset shape — what each subject brought to the scanner — is visible at a glance.
  • Multi-modality lands gracefully via icons, not tabs.
  • Modern, inviting; reduces onboarding friction without dumbing down.
  • Editor reuses the same shell — minimal user-mental-model switch.

Weaknesses

  • Card grid doesn't scale visually past ~30 subjects; needs pagination or a "compact list" toggle.
  • Card-as-unit hides cross-subject patterns (e.g., "all bold rows missing tasks").
  • Custom widget work — not a stock Qt layout.

Notes

  • Cards via a custom QWidget in a QScrollArea + QGridLayout; not a model/view pattern.
  • The card layout is reusable as a "drill-in" view for Inspector and Grid proposals.
  • ⌘K palette would be a real implementation effort but pays for itself across both views.

Best for: mid-size studies (5–30 subjects) where users care about seeing what each subject brought before deciding how to convert. Highest design effort, highest "wow" factor; pair with a "compact list" toggle for scale.

1 What you're looking at

  1. 1Filter chips at top. Click-built (faceted dropdowns) or typed. Multiple filters compose. Saved filters are first-class. datatype = func + status ≠ ok + subject = sub-002 demonstrates real query power.
  2. 2Bulk-edit bar. Visible only when rows are selected. Pick a column and apply once across N rows.
  3. 3Side rails. Filter, raw FS, BIDS preview — all collapsible. Same on both Converter and Editor.
  4. 4Inline cell editing. Tab into a cell, type, schema validates on every keystroke; errors paint the cell red.
  5. 5Detail panel (right). Properties / Why / Source sub-tabs. Properties is the entity-template form; Why is the classification rationale; Source shows raw DICOM headers.
  6. 6Editor view. Same shell. The grid switches to file rows instead of recording rows. Detail panel becomes the type-routed viewer (sidecar form, NIfTI, TSV) plus the file's validation messages.

2 User journey

  1. Filter. Add chips for "datatype = func" + "status ≠ ok" → focuses on the rows that need work.
  2. Bulk fix. Multi-select 3 bold rows missing task; bulk bar appears; type rest once; all three update.
  3. Group by. Group by subject for review; group by datatype for batch fixups.
  4. Inspect. Click ⋯ to open Why? sub-tab; see the classifier rationale.
  5. Run. Toolbar Run; same dry-run-diff confirmation.
  6. Editor. Grid now lists generated files. Filter status ≠ ok to focus on broken sidecars; click a file to see the schema-driven form on the right; one-click fix.

Strengths

  • Scales to thousands of rows — only proposal that handles 200-subject studies comfortably.
  • Bulk editing is fast and obvious.
  • Filters + group-by give real query power without SQL syntax.
  • Maps onto QTableView + custom model — efficient Qt implementation.
  • Same shell for Converter and Editor — minimal context switch.

Weaknesses

  • Cell-by-cell editing hides multi-field rules (changing datatype changes valid entities).
  • Provenance is one tab away; easy to forget the "why" behind a classification.
  • Visually colder; harder to use as a teaching tool.

Notes

  • Best technical fit for current Qt expertise: standard model/view, no custom drawing.
  • Inline cell editor needs to switch widget per entity (combo for datatype, regex-validated input for label entities, integer spinner for index entities).
  • Could ship as the default + Inspector mode as an opt-in panel.

Best for: labs with 50+ subjects per project where curation is a daily activity. Most pragmatic engineering choice; lowest risk of over-design.

1 What you're looking at

  1. 1Row list (left). Always-visible compact list of every recording (Converter) or every file (Editor). Status badge + confidence + brief metadata. The "skim" surface.
  2. 2Raw / file pane (middle). Converter shows the raw acquisition: file location, full DICOM headers, inferred identity with evidence. Editor shows the file content with type-routed viewer (NIfTI tri-planar, sidecar form, TSV table, raw DICOM dump).
  3. 3Schema engine arrow. Visually anchors the transformation: in Converter raw → BIDS, in Editor file ⇄ schema rules. Hovering reveals the exact rules that fired.
  4. 4BIDS / validation pane (right). Converter shows predicted BIDS layout, "why this name?" rationale, sidecar field preview with REQ/REC/OPT colour, edit form at the bottom. Editor stacks file/folder/dataset validation with one-click fixes.
  5. 5Top toolbar prev / next. Walk through every row sequentially; no card-counting. Useful for QC at the per-recording level.

2 User journey

  1. Pick subject in left list. Middle pane fills with raw evidence; right pane fills with predicted BIDS.
  2. Read the why. Rationale is on screen — no clicking around. Disagree? Edit the entity form at the bottom of the right pane.
  3. Walk through. next-row button cycles every recording; QC pass takes minutes for a 30-row dataset.
  4. Pre-conversion sidecar audit. Right pane already shows what fields will be filled, missing, or __TODO__. Catch problems before running.
  5. Run all. Same dry-run-diff confirmation; conversion proceeds.
  6. Editor (post-conv). Same shell; middle becomes type-routed viewer (NIfTI for .nii.gz, schema form for .json, TSV table for physio.tsv.gz); right side stacks file/folder/dataset validation.

Strengths

  • Pedagogically strongest — users learn BIDS by using it.
  • Provenance front and center — never buried in a context menu.
  • Sidecar audit before conversion (most tools do this after, if at all).
  • Editor view inherits the same arrow-anchored shell.
  • One-row-at-a-time pacing matches QC workflows in audit-heavy labs.

Weaknesses

  • Sequential pacing is slow at scale; needs a "list view" toggle for 200-subject runs.
  • Two text-heavy panes can fatigue at length.
  • Bulk operations don't fit naturally; need a separate mode.

Notes

  • Split-screen widget reusable in any other proposal as a "drill-in" detail view.
  • Pairs naturally with Inspector or Grid as the "focus on one row" detail mode.
  • The provenance machinery this requires is needed regardless (architecture.md §10).

Best for: teaching environments, QC-heavy labs, anyone who has to defend why a row was classified the way it was. Strongest as a complement to Inspector or Grid for power users who occasionally need to drill in.
